## Findings considered and rejected

Recorded so nobody re-audits them. (Category tags are from the audit pass.)

- **[BUG-01] falsy checks conflate 0 with null after `coerceInt`**
  (`packages/js/src/athlete-set-write.ts:235,284,520,617`): real pattern, but
  TrainHeroic ids are positive integers — id 0 does not occur upstream, so the guard is
  unreachable in practice. Not worth an executor run.
- **[SEC-01] distinct login errors (allowlist-denied vs bad credentials)**
  (`packages/cloudflare/src/auth/handler.ts:150,157`): deliberate — the two paths carry
  separate Sentry metrics and the "not permitted" message is intentional UX for private
  deploys. Leaks only allowlist membership, not upstream account existence.
- **[SEC-02] claimed rate-limit bypass via authorize redirect**: misreads the flow —
  every credential-bearing request (`/token`, `/register`, POST `/authorize`) hits the
  tight `LOGIN_RATE_LIMITER` before any other work (`src/index.ts:45-58`).
- **[SEC-03] account-registry last_seen refresh**: best-effort observability table,
  never gates access — by design per the code comment.
- **[SEC-04] credentials in OAuth grant props**: documented architecture (upstream API
  has no refresh token); not actionable.
- **[PERF-01] linear scan in `ExerciseLibrary` search**
  (`packages/js/src/exercise-index.ts:148-167`): O(N) over an in-memory map of a few
  thousand rows is microseconds per query; the hosted path already filters in SQL.
- **[PERF-02] roster-activity per-athlete fan-out** (`packages/js/src/athlete.ts:108`):
  documented upstream limitation ("There is no single roster-activity endpoint") with a
  documented mitigation (pass a subset); a TTL cache would add staleness for little.
- **[DEBT-02] duplication between `ExerciseLibrary` and `ExerciseStore`**: the shared
  parts (`buildSearchText`, `rankSearch`, `withUnits`, presentation) are already
  extracted; the residue is inherent to Map-iteration vs SQL `LIKE`, and a forced common
  engine would defeat SQL-side filtering.
- **[DEBT-03] drizzle-orm pinned to 1.0.0-rc.3**: the audit claimed 1.0 stable exists;
  it does not (npm `latest` is 0.45.2; 1.0.0 has only beta/RC tags). The RC pin is
  documented in `packages/cloudflare/CLAUDE.md` and is *ahead of* stable. Revisit when
  1.0.0 final ships.
- **[DEPS-01] "Node >=24 too aggressive"**: Node 24 is an active LTS line; the engines
  requirement is documented in CLAUDE.md and enforced in CI. Stale-knowledge finding.
- **[DEPS-02] "MCP SDK 2.0 churn coming"**: speculation with no release signal; not a
  finding.
- **[DEPS-03] "TypeScript 6.0.3 too new, downgrade to 5.x"**: the workspace typechecks
  and ships on TS 6 in CI; downgrading has real cost and no identified benefit.
  Stale-knowledge finding.
- **[DX-01] missing `.env.example`**: the toolkit needs exactly two user env vars,
  documented prominently in the README (which explicitly warns against putting them in
  files); an `.env.example` would invite the pattern the README warns about.
- **[DOCS-01] website tool index can drift**: it can't in practice — `predev`/`prebuild`
  run `gen:mcp-tools` on every build, and the generator asserts list/summary parity.
