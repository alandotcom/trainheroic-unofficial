athlete bank (13 queries) and the full coach bank (8 queries) through the local stdio servers,
read-only. This run reflects the v0.6.0 code the servers booted with; the fixes listed under
"Shipped this round" land in core and need a server reconnect before they show up in an eval.

The logged-vs-prescribed group is now effectively solved: every "did I record / what did I do /
did I actually do my squats" query resolved in 2 calls at confusion 1, off the `loggedOnly` flag
and the `performed`/`logged` fields. That was the historically worst group.

Remaining athlete friction, by theme:

- **All-time aggregates went the long way (confusion 4, worst run).** "How many sessions
  all-time" produced 18 calls, three HTTP 504 timeouts on wide windows, and a *wrong* hand-tally
  (310 vs the real 343). The agent never reached for `athlete_profile`, which returns
  `summary.sessions_count` (343) plus total reps/volume and first/last logged in one call.
- **"Next workout" is ambiguous when nothing is scheduled forward (confusion 2, partial).** The
  agent queried today→+7, got nothing, and had to widen backward to find yesterday's unlogged
  session. No guidance to do that.
- **Per-exercise history has no date filter.** Four runs (back squat, bench trend, most-squatted,
  bench PR) noted that `athlete_exercise_history` returns the full all-time series (49 sessions
  for back squat) and made the model filter in-context; several asked for a `since`/`until`.
- **Dense multi-program weeks trip the result budget.** "Schedule this week" offloaded to a file
  and needed a Python parse even with `limit`; `limit` caps the presented count but does not stop
  transport-layer truncation on data-heavy sessions.
- **Minor:** `athlete_working_maxes` can return a null value (slot exists, no number) with no
  explanation; `athlete_exercises` ignored `limit` when no `q` was passed (returned 200+ on
  `limit:10`); estimated1RM from a rep formula can read as a real single.

Recurring coach friction:

- **Team headcounts read as contradictory (confusion 3).** `member_count` and `athlete_count` are
  both 0, `athleteIds` holds only the owner, and `list_athletes` returns 3. On this account the
  counts are actually right — the demo athletes are org-roster but not enrolled team members — but
  nothing said so, and there is no per-athlete team field to attribute the roster.
- **No coach-facing per-athlete PR lookup (confusion 3).** `athlete_personal_records` lives in the
  athlete namespace and reflects the authenticated user; it only "worked" because this coach shares
  athlete scope with the same person. A coach asking for an arbitrary roster athlete's PRs has no
  tool. "Recent" PRs also can't be honored — the PR board has no dates.
- **Team-wide volume is per-athlete.** Agents queried `training-summary-athlete` one id at a time
  (it already accepts many ids in one call — they didn't know). The raw catalog also exposes a
  `training-summary/teams` endpoint that `analytics_query` doesn't wire up; verified it returns 200
  for `{teamId, date}` but rows are empty here because the team has no enrolled members.
- **`list_programs` empty → "no programs".** Already cross-referenced, still reads as a dead end;
  the team's `group_program` id has no name inline.
- **Minor:** `can_edit` (custom-exercise marker) undocumented; `messaging_read` `isAuthor`
  undocumented and no `since`/last-message summary; `daysSinceLastLogin` implies training activity.

## Shipped this round

Description tuning plus small logic fixes and one new coach tool. `pnpm check` green; logic
changes verified against the live API and re-eval'd via the standalone runner (see the
confirmation table below).

- **`athlete_profile`** description now leads with the all-time session count and tells the model
  to use it for "how many sessions / total volume ever" instead of summing windows.
  `athlete_workouts` gained a matching pointer ("date-windowed fetch, NOT an aggregate; a
  multi-year range can time out").
- **`athlete_exercise_history`** gained `since`/`until` (YYYY-MM-DD, inclusive) filtering on the
  session time-series (PRs stay all-time), with the description steering "last 3 months / this
  year" questions to it.
- **`athlete_workouts`** description: "next workout" guidance (query forward; if empty, widen back
  a day or two for the unlogged session) and a note that `loggedOnly` filters at the session level.
- **`athlete_exercises`** `limit` now applies when no `q` is passed (was silently ignored).
- **`athlete_working_maxes`** documents the null-value case.
- **`analytics_query`** documents that `userIds` takes many athletes in one call (verified: a single
  call returned all 18 rows across both demo athletes), so team-wide volume is one request, and that
  an athlete with no sessions returns no rows.
- **`list_teams`** documents member_count/athlete_count (enrolled, often 0) vs athleteIds (owner
  only) and points at `list_athletes` for the roster.
- **`list_athletes`** documents that it is the org-wide roster across teams, `daysSinceLastLogin` is
  login not training, and demo athletes are included.
- **`list_programs`** description strengthened to push toward `list_teams` group-programs on empty.
- **`exercise_search` / `exercise_resolve`** document `can_edit` (1 = your custom exercise) and when
  to prefer resolve.
- **`messaging_read`** documents `isAuthor` (true = you sent it) and the empty-thread meaning.

Also shipped:

- **Standalone eval runner (`scripts/mcp-eval.sh`).** Runs one query through a fresh headless
  `claude -p` that spawns its own stdio server from `.mcp.json`, so every run loads current source
  with no reconnect and no dependency on the session's pinned MCP connection. Read-only by
  construction (read tools whitelisted, writes denied). The `mcp-eval` skill now documents it as the
  primary path. This removes the "reconnect the servers after a code change" friction entirely.
- **Coach `athlete_lift_history(athleteId, exerciseId, since?, until?)`.** The coach-side per-roster
  athlete PR board + dated session series (verified the coach token reads `/v5/exercises/{id}/
  history?userId=X`, which embeds dated liftPRs; `personalRecords` can't be parameterized). Closes
  the "no coach tool for an athlete's PRs" gap that previously only worked by shared athlete scope.
- **`historyInRange` helper** (`packages/core/src/history.ts`) backs both the athlete history date
  filter and the new coach tool; covered by `packages/core/test/history.test.ts` (5 cases).

## Next (not yet done — structural, candidates for the next iteration)

- **Cross-exercise PR discovery for an athlete.** `athlete_lift_history` is per-exercise, so "show
  me their PRs" still requires guessing which lifts to query (confusion-3 driver). An
  `athlete_pr_summary` / `athlete_logged_exercises(athleteId)` would let a coach discover lifts
  instead of guessing. Needs an API path that lists a roster athlete's logged exercises
  (`/v5/users/exercises/history?userId=X` ignores the userId and returns the caller's own).
- **Per-team roster.** A `list_team_athletes(teamId)` or a `teamIds` field on `list_athletes`, so
  "athletes per team" is one unambiguous call instead of cross-checking counts.
- **`athlete_next_workout` / `nextUnlogged`** convenience so "what's next" is one call, plus a
  `programs` filter for multi-program athletes.

- **Coach per-athlete PR/history tool.** A coach-namespace way to pull a named roster athlete's PRs
  or recent lifts (likely via the lift analytics endpoints, since PRs are user-token scoped). This
  is the strongest unaddressed confusion-3.
- **All-time/aggregate ergonomics.** A `countOnly`/summary mode on `athlete_workouts`, or simply the
  `athlete_profile` pointer (shipped) proving enough — re-eval will tell.
- **Wire `training-summary/teams` into `analytics_query`** (and possibly `lift-one-rep-max/teams`).
  Endpoint verified to respond; needs a non-empty team to confirm row semantics and the single-date
  vs range param before shipping.
- **Surface the team's group-program name** in `list_teams`/`get_team` output (resolve the id).
