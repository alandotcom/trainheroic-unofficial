# MCP eval — 2026-06-21 (round 3c): roster_activity

Third loop iteration. After round 3b, the "most recently active athlete" run still fanned out to
analytics just to rank recency, because `list_athletes` only carries `daysSinceLastLogin` (app
login, not training). Both that run and "anyone falling behind" wanted a per-athlete
last-training-date.

## Where the recency signal lives

No single roster-activity endpoint surfaced (the coach home feed is a cross-origin
`adapter.trainheroic.com` iframe; the coachapp athletes route redirected away). But the data was
already reachable: `GET /v5/athleteProfile/summary?user_id={athleteId}` returns `sessions_count`,
`first_logged_date`, and `last_logged_date`, and the coach token honors it for a roster athlete.

## Shipped

- **`roster_activity(athleteIds, useMetric?)`** (coach core tool): each athlete's all-time
  snapshot (sessionsCount, first/last logged date, total reps/volume), sorted most-recently-active
  first. Backed by `fetchRosterActivity` in `js`, which fans the summary lookup out per athlete with
  a small concurrency cap (no single endpoint exists) and normalizes the API's epoch placeholder
  (`1970-01-01`, returned for a never-logged athlete) to `null`. Ties on recency break by session
  count. Defined in `core`, so local and hosted both get it. Tested (`sortRosterByRecency` +
  `fetchRosterActivity` with a stub client).
- **`list_athletes`** points at `roster_activity` for "most recently active / falling behind",
  steering away from `daysSinceLastLogin`.

"Most recently active" is now a clean three-call chain: `list_athletes` → `roster_activity` (ranks
by lastLoggedDate) → `athlete_training` for the top athlete. The re-eval also caught two bugs in the
first cut of `roster_activity` (epoch date instead of null; no tiebreaker), both fixed before ship.
