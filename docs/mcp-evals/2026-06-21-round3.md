The coach surface is clean except for two outliers, and both are the same root cause: **there is no
way to discover which exercises a roster athlete has logged.** "Most recently active athlete's
training" (12 calls, confusion 4) and "Kyle's recent PRs" (13 calls, confusion 3) each guessed
exercise ids one at a time and fired a per-exercise history call for each. The first run wasted eight
calls on an athlete (the coach's own account) who has zero logged sessions, because nothing said so
up front.

I probed the live API hard for a discovery endpoint and confirmed none exists for the coach token:
`/v5/users/{id}/exercises/history`, `/v5/athletes/{id}/exercises`, `/v5/athletes/{id}/personalRecords`,
a coach feed, and a per-team athlete roster all 404; `/3.0/athlete/programworkout/range` ignores a
`userId` param (it returns the coach's own calendar either way); `/v5/users/exercises/history?userId=X`
ignores the id and returns the caller's own (empty for the coach). Only `/v5/exercises/{id}/history?userId=X`
(per-exercise) and the analytics report pulls honor a roster athlete id. So cross-exercise discovery
cannot be a new live tool; the round-3 fix is to stop the blind fan-out with guidance, and to move
the real rollup onto the D1 warehouse (see "Next").

- **`list_athletes` has no training-recency field.** `daysSinceLastLogin` is login, not training, so

## Shipped this round

Description tuning plus two code changes and one new presenter. `pnpm check` green; the logic
changes were verified against the live API and re-eval'd via the standalone runner (table below).

- **`athlete_workouts` summary mode.** New `summary:true` returns one compact row per session
  (`id, date, title, program, team, logged, exerciseCount, performedCount`) instead of every set,
  via a new `summarizeAthleteWorkouts` presenter in `js` (covered by two tests). The description
  steers overview questions ("what's on my schedule", "what have I been training") to it and notes
  to read `performedCount` against `exerciseCount` (e.g. 1 of 12 logged).
- **`list_athletes` token matching.** The q filter now requires each whitespace-separated word to
  appear in the record independently, so "Kyle Jones" matches "Jones, [Demo] Kyle". The description
  documents the behavior, that the account owner is in the roster, and that per-team attribution
  comes from each athlete's `groups`/`groupTitles`/`teamCount` (there is no per-team roster endpoint).
- **`athlete_lift_history` discovery triage.** The description now tells the coach not to fan out
  blind per-lift calls: first run `analytics_query metric:training-summary-athlete` for the athlete;
  no rows means no logged training in range, so stop; rows mean query the standard barbell lifts.
- **`analytics_query`** documents `training-summary-athlete` as the per-athlete "has this athlete
  trained at all, and when" triage, and that it accepts 1 to N userIds despite the singular name.
- **`athlete_exercise_history`** warns that estimated1RM inflates on a mixed heavy-single +
  high-rep-backdown session, and notes that `liftPRs` each carry a date so "PRs set this year" can be
  filtered client-side.
- **`athlete_exercises` / `athlete_personal_records`** document the variant split (a common name
  returns several variants, each with its own board; prefer the plain canonical entry, check others
  if a PR looks missing).

## Next (candidates for the next iteration)

The strongest unaddressed items now point at the D1 warehouse rather than the live API. Because the
hosted Worker already syncs the athlete's own training into D1 (`athlete_pr` holds every exercise's
PR board, `athlete_exercise_session` holds dated per-exercise sessions, `athlete_workout` holds the
session headers), we can build projections the live API will not serve cheaply. These are
hosted-only (the local stdio servers have no D1), so they are eval'd with `target=hosted` after a
sync, not by the default local run.

- **Cross-exercise PR rollup / best-lifts (athlete, D1).** `GROUP BY exerciseId` over `athlete_pr`
  joined to `athlete_exercise` answers "most I've ever squatted", "all my PRs", and "best lift" in
  one query, collapsing the variant fan-out that "most squatted" and "bench PR" still pay. This is
  the rollup both runs explicitly asked for.
- **Logged-exercise index with frequency + recency (athlete, D1).** `COUNT(*)` and `MAX(date)` over
  `athlete_exercise_session` per exercise gives the usage signal the live catalog lacks (its 349
  rows carry no counts), so variant disambiguation becomes data-driven ("which Bench Press do I
  actually use").
- **Coach roster-athlete discovery stays API-bound.** The warehouse is self-scoped and there is no
  coach endpoint to bulk-ingest a roster athlete's history, so the cross-exercise gap for a named
  athlete cannot move to D1. The triage guidance shipped this round is the right local mitigation.
- **`exercise_resolve` ranking.** Prefer higher `use_count` so a zero-use branded variant does not
  outrank the common lift.
- **Session-date drill-down for a roster athlete.** A coach tool that, given an athlete and a date,
  returns the exercises logged that day would turn "Kyle's Clean Focus session" into the right
  exercise id without guessing. Needs an endpoint that honors the athlete id for a saved workout;
  not yet found.
