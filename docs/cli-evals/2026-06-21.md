# CLI eval ‚Äî 2026-06-21 (Haiku, new coach/athlete surface)

First run of the `cli-eval` skill, aimed at the surface added this session: the coach
"Log for Athlete" flow plus full coach-command parity (`roster-activity`,
`athlete-training`, `athlete-lift-history`, `athlete-workouts`, `log-set`, team/team-code
CRUD, `analytics-query`, session lifecycle) and `athlete workouts --summary`/`--log-ids`.

Goal (from the session): drive the **entire CLI surface on Haiku** and **loop, fixing
confusion between rounds, until it bottoms out**. Model held at Haiku throughout ‚Äî a weaker
model leans hardest on command names / flags / help text / output shape, so it surfaces the
most fixable confusion. 11-query bank: 5 reads + 6 writes (writes on the TEST account).

## The drivers and the fixes

Ranked by frequency √ó severity in round 1. Each fix re-measured in a later round.

1. **`log-set` id discovery (the dominant write confusion).** `--set` wants a
   `savedWorkoutSetId`, but `--raw` output carries 4+ id fields (`workout_id`, `set_id`,
   `workout_set_exercise_id`, `saved_workout_set_id`, ‚Ä¶) and nothing said which. AW1 tried
   four wrong ids and got the same opaque `not found on this date` each time ‚Üí score 5.
   - **Enriched the not-found error** (`findSavedWorkoutSet`, `js/src/athlete.ts`) to list
     the `savedWorkoutSetId`s present on that date *and* each set's exercise ids; empty ‚Üí
     "no workout scheduled on this date." AW1 **5 ‚Üí 2** (one wrong-id attempt, then the error
     told it exactly which id to use).
   - **Added a `--log-ids` projection** (`presentLogTargets` in `js`, flag on both
     `coach athlete-workouts` and `athlete workouts`) emitting just
     `{date, savedWorkoutSetId, exercises:[{savedWorkoutSetExerciseId, title, units,
     prescribed, performed}]}`. Replaces grepping a ~98 KB blob. CW1/AW1 then logged on the
     **first attempt** with zero id-guessing.
2. **Demo-athlete 401 masked as "not found."** Seeded demo athletes are read-only for
   results; the write 401s, but the agent first failed to even locate a set and saw a generic
   error. **Named the cause** on a 401/403 coach write ("‚Ä¶appears to be read-only for results
   ‚Äî seeded demo/sample athletes return 401; results only persist for real invited athletes").
   CW2 **4 ‚Üí 2** (now a clear, documented blocker).
3. **`analytics-query` metric names ‚âÝ `coach analytics` categories.** Agents read the raw
   `/v5/analytics` catalog, tried those names, got rejected; per-metric required params were
   undocumented. **`analytics-query` with no `--metric` now prints a catalog**
   (`analyticsMetricCatalog()` in `js`) of scope + required/optional params, HELP notes the
   keys are curated (not the raw categories), and signposts that **team training volume lives
   in `roster-activity --metric`**, not analytics-query. CR2 **3 ‚Üí 2**.
4. **`athlete-training` empty looked broken.** Reached for as "how has X trained," returns
   `{sessions:[]}` when nothing was logged that month. **Empty-result `note`** added (also for
   `athlete-lift-history`) distinguishing "nothing logged" from an error, plus HELP that
   frames the three reads as distinct lenses (training = logged-by-month, athlete-workouts =
   prescribed+logged over a range, lift-history = one exercise).
5. **`coach athlete-workouts` lacked `--logged-only`/`--summary`** that `athlete workouts`
   has, forcing eyeballing of a 78 KB payload. **Added both** for parity.
6. **Minor HELP wins:** `team-code-delete --code` is the id from `team-code-create` (CW4
   "preemptively addressed"); `log-set --date` is the workout's *scheduled* date, not today;
   athletes log to *prescribed* workouts only.

## Test-data artifact (the lever the account itself provided)

CR3/CR4/AR1 kept scoring 3 across synthetic rounds for a reason unrelated to the CLI: the
test athlete (test athlete, 200004) had **one logged session, dated 2026-06-22** ‚Äî a day *past*
the eval's "today" (06-21). Every sensible "recent workouts" query ended at 06-21 and found
nothing until the agent discovered the future date via `roster-activity`.

Fix was to the **data, not the code**: logged a realistic ~4-week block against the existing
prescribed sessions (Back Squat 275‚Üí285‚Üí295 across 06-24/06-29/07-08, plus bench, deadlift,
OHP ‚Äî 9 logged sessions, 06-15‚Ä¶07-08, with partial completion for "did I do X vs just
schedule it" contrast), then re-ran reads as-of **2026-07-12**. CR3 **3 ‚Üí 2** (17 cmds ‚Üí 4),
AR1 **3 ‚Üí 1** (one command). Enrichment batch: `/tmp/enrich-alan.sh`.

## Harness fix (found while sanity-checking)

The MCP-eval runner (`scripts/mcp-eval.sh`) granted the subagent bare `Bash`, so an
MCP-under-test agent could shell out to the `trainheroic` CLI and answer there ‚Äî the athlete
sanity run did exactly that, resolving via the CLI instead of MCP. Tightened both runners so
each surface sees only its own tools: `mcp-eval.sh` allow-list is MCP tools only (no `Bash`,
no `Read`); `cli-eval.sh` drops `Read` (it already had an empty MCP config), leaving
`Bash(trainheroic:*)` only.

## Residual (not worth chasing in code)

- **Ad-hoc athlete logging** (AW1): an athlete can only log to a *prescribed* workout; "log
  today's session" with nothing scheduled is unfulfillable. A real feature gap
  (`athlete create-unplanned-session`), now at least surfaced in HELP.
- **Date-windowed team volume** (CR2): `roster-activity` is an all-time snapshot; "last two
  weeks" isn't directly answerable. HELP now says all-time + ids from `coach athletes`.
- **Haiku noise**: ¬±~0.5/run with no code change (AR1/CW4 bounced 1‚Üî2 across rounds).
